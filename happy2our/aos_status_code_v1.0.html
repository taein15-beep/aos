v0.2 보완사항 (2026-06-09 추가)

  • [결제상태] PAY_EXPIRED(결제만료) 정식 상태 추가 — PAY_PENDING 유효시간 초과 시 자동 만료, 재결제 시 신규 PAY_PENDING 생성
  • [상품상태] PRD_CLOSED와 좌석상태 SEAT_CLOSED 역할 명확히 구분 — 특정 출발일 좌석 마감이 상품 전체 PRD_CLOSED를 의미하지 않음
  • [예약상태] RES_CONFIRMED 정산 영향을 "정산예정"으로 수정 — 실제 정산 확정은 결제상태·출발완료 기준과 함께 판단
  • [결제상태] PAY_*가 예약 단위 결제 요약상태임을 명시 — 개별 결제 거래는 추후 payment_transaction 상태값으로 분리 가능

개요 상태값 설계 원칙 및 코드 체계
본 문서는 AOS 플랫폼 전반에서 사용되는 표준 상태값(Status Code)을 정의합니다.
v0.2에서는 예약상태와 결제상태의 완전 분리, 상품공유와 노출상태 분리, 알림톡 템플릿 승인상태 추가, 지자체 지원사업 흐름 고도화를 반영했습니다.
모든 상태값은 도메인접두어_영문대문자 형식을 따르며, 개발자가 임의로 추가·변경할 수 없습니다.

📐 상태 코드 명명 규칙

  • 형식: {DOMAIN}_{STATUS}
  • DOMAIN: 2~5자 영문 대문자 접두어
  • STATUS: 의미를 명확히 표현하는 영문 대문자
  • 구분자: 언더스코어(_) 단일 사용
  • 최대 길이: 30자 이내 / DB: VARCHAR(30) NOT NULL
  • 변경 제어: Enum 또는 코드 테이블로 서버사이드 통제
  • 이력 관리: *_status_history 테이블 전량 기록

🏷 도메인 접두어 전체 목록 (v0.2)

  • PRD_ — 상품(Product)
  • SHARE_ — 상품공유(Product Share)
  • EXPO_ — 판매사 노출(Exposure) ★신규
  • RES_ — 예약(Reservation)
  • PAY_ — 결제/환불(Payment)
  • DEP_ — 출발(Departure)
  • SEAT_ — 좌석(Seat)
  • STL_ — 정산(Settlement)
  • INV_ — 인보이스(Invoice)
  • MSG_ — 메시지발송(Message)
  • TPL_ — 알림톡템플릿(Template) ★신규
  • GOV_ — 지자체지원사업(Government)
⚠ v0.2 핵심 설계 원칙 — 반드시 숙지
  • 예약상태(RES_*)와 결제상태(PAY_*)는 완전히 독립된 컬럼으로 관리한다. 예약확정(RES_CONFIRMED)은 담당자 확정 기준이며, 결제 완료(PAY_PAID) 여부와 무관하다.
  • 상품공유 승인(SHARE_ACCEPTED)이 홈페이지 노출을 의미하지 않는다. 판매사가 카테고리 연결 후 노출 체크를 완료해야 EXPO_VISIBLE이 된다.
  • PRD_CLOSED와 SEAT_CLOSED는 다르다. SEAT_CLOSED는 특정 출발일(회차) 마감이고, PRD_CLOSED는 상품 전체 마감이다. 특정 출발일만 마감된 경우 PRD_CLOSED로 전환하지 않는다.
  • PAY_EXPIRED 발생 시 기존 결제 건을 재활성화하지 않는다. 재결제 원할 시 신규 PAY_PENDING을 생성한다.
  • PAY_* 상태는 예약 단위 결제 요약상태다. 개별 결제 거래 상세는 추후 payment_transaction 상태로 분리 가능하다.
  • 신규 상태값 추가 시: 기획팀 승인 → 백엔드 리드 검토 → 본 문서 업데이트 → 코드 테이블 반영 순서를 반드시 준수한다.
  • 모든 상태 전이 유효성 검증은 서버사이드 전용. 클라이언트 파라미터 직접 수용 금지.

03 예약상태 (Reservation Status) v0.2 전면 재정의
v0.2 핵심 변경사항:
① 예약상태에서 환불 관련 상태(RES_REFUNDING, RES_REFUNDED, RES_REFUND_FAILED) 제거 → 결제상태(PAY_*)로 이관

💡 PAY_* 상태와 payment_transaction 관계

PAY_* (결제상태): 예약(reservation) 단위의 전체 결제 진행 요약상태. 예약금 납부 여부, 잔금 납부 완료 여부, 환불 진행 상태를 표현.
payment_transaction (개별 거래): 카드 승인 1건, 가상계좌 입금 1건, 무통장 입금 1건 등 실제 개별 결제 처리 단위. PAY_*에 1:N으로 연결.
현재는 PAY_* 단위로 관리하며, 결제 수단별 트랜잭션 상세 추적이 필요한 시점에 payment_transaction 상태값(TXN_PENDING, TXN_APPROVED, TXN_FAILED 등)으로 분리 설계 가능합니다.

╔══════════════════════════════════════════════════════════════════════════╗ ║ 예약상태(RES_*)와 결제상태(PAY_*)는 완전히 독립적으로 움직인다 ║ ╚══════════════════════════════════════════════════════════════════════════╝ ┌─────────────────────────────────────────────────────────────┐ │ 예약상태 흐름 (RES_*) │ └─────────────────────────────────────────────────────────────┘ [고객 예약 폼 제출] │ ▼ RES_RECEIVED ──────────────────────────────────► RES_EXPIRED (예약접수) (담당자 미처리·기간 초과 자동) (예약만료, 최종) │ │ [담당자 검토 필요 시] ▼ RES_WAITING ──────────────────────────────────► RES_EXPIRED (예약대기) │ │ [담당자가 좌석·상품·출발 가능 확인 후 확정] │ (결제 완료 여부와 무관) ▼ RES_CONFIRMED ◄──── RES_WAITLISTED ←── RES_RECEIVED (예약확정) (대기예약) (좌석 마감 시) │ │ [고객 취소 요청] ▼ RES_CANCEL_REQUESTED ──► [담당자 취소 처리] (취소요청) │ 또는 담당자 직접 취소 ▼ RES_CANCELLED ─────────────────── 환불은 PAY_* 에서 진행 (예약취소, 최종) RES_CANCELLED ≠ 환불완료 RES_CONFIRMED ──[DEP_COMPLETED 연동]──► RES_COMPLETED (행사완료, 최종) ┌─────────────────────────────────────────────────────────────┐ │ 결제상태 흐름 (PAY_*) │ │ (예약상태와 독립적으로 별도 관리) │ └─────────────────────────────────────────────────────────────┘ [RES_RECEIVED 생성 시 자동 생성] │ ▼ PAY_UNPAID ──────────────────────────────── 결제 안내 알림톡 발송 (미입금) │ [결제 링크·가상계좌 발행] ▼ PAY_PENDING ──────────────────────────────► PAY_FAILED (재시도 가능 → 신규 PAY_PENDING) (결제대기) │ │ │ [유효시간 초과]│ │ ▼ │ PAY_EXPIRED (최종, 시스템 자동) │ → 재결제 원할 시 신규 PAY_PENDING 생성 │ │ [예약금만 납부] [전액 납부] ▼ ▼ PAY_DEPOSIT_PAID PAY_PAID (예약금입금) (결제완료) │ │ │ [잔금 기한 도래] │ ▼ │ PAY_BALANCE_REQUESTED │ (잔금요청) │ │ [잔금 납부] │ └───────────────────────────────┘ │ ▼ PAY_PAID (결제완료) ─── 취소/환불 흐름 (PAY_* 내부에서 처리) ───────────────────────── PAY_DEPOSIT_PAID PAY_PAID PAY_BALANCE_REQUESTED ──► PAY_REFUND_REQUESTED (고객·운영자 요청) PAY_PAID │ │ [담당자 승인] ▼ PAY_REFUNDING (환불중) │ ┌───────────────┼───────────────┐ │ │ │ [전액환불] [부분환불] [실패] │ │ │ PAY_REFUNDED PAY_PARTIAL_REFUNDED PAY_REFUND_FAILED (환불완료) (부분환불, 수수료 차감) (수동처리 필요) (최종) (최종)
💡 핵심 설계 원칙: RES_CONFIRMED(예약확정)은 담당자 확정 기준이며 결제 완료와 무관합니다. 예) RES_CONFIRMED + PAY_DEPOSIT_PAID = "예약확정 상태이지만 잔금 미납" → 화면에 "예약확정 | 잔금미납" 조합 표시 예) RES_CANCELLED + PAY_REFUNDING = "예약은 취소됐으나 환불 처리 중" → 예약취소와 환불은 별도 상태로 추적

06 지자체 지원사업 흐름도 (v0.2 고도화)
[여행사가 지자체 공모사업 확인] │ ▼ GOV_DRAFT ──────────────────────────────► GOV_CANCELLED (최종) (사업등록) │ │ [신청 서류 준비 완료] ▼ GOV_READY ──────────────────────────────► GOV_CANCELLED (최종) (신청준비) ◄──────────────────────── GOV_REJECTED (재신청 시 복귀) │ │ [공모 신청 제출] ▼ GOV_APPLIED ────────────────────────────► GOV_CANCELLED (최종) (신청완료) ← 지자체 심사 대기 │ ├──[미선정·반려]──► GOV_REJECTED ──► GOV_READY (재신청 준비) │ (신청반려) 또는 포기 시 GOV_CANCELLED │ │ [지자체 승인, 사업 시작일 도래] ▼ GOV_ACTIVE ─────────────────────────────► GOV_CANCELLED (최종) (진행중) · 할인코드 활성화 · 예약 시 자동 할인 적용 · 예산 소진 모니터링 (80% → 경고) │ │ [사업 기간 종료] ▼ GOV_REPORTING (결과보고중) · 참가 인원·집행 금액 집계 · 증빙 서류 첨부 (영수증, 명단) · 결과 보고서 작성·제출 │ │ [보고서 제출 완료] ▼ GOV_SETTLING (정산중) · 지자체 전용 정산서 생성 · 증빙 서류 최종 제출 · 사업비 정산서·세금계산서 처리 │ │ [정산 완료] ▼ GOV_COMPLETED (최종) · 최종 정산서 확정 저장 · 사업 실적 아카이빙
💡 GOV_ACTIVE 상태일 때만 할인코드 활성화. 예산 소진 시 시스템이 자동으로 할인 코드를 비활성화하고 관리자에게 알림을 발송합니다.

규칙 공통 구현 규칙 및 개발 가이드

🗄 DB 설계 가이드

  • 모든 상태 컬럼: VARCHAR(30) NOT NULL
  • 상태값 제약: CHECK CONSTRAINT 또는 Enum
  • 이력 테이블: *_status_history
  • 이력 필드: from/to/changed_by/changed_at/reason
  • 소프트 삭제: is_deleted + deleted_at
  • 좌석 선점: version 컬럼 필수 (낙관적 잠금)
  • 예약-결제 관계: 1:N (예약금+잔금 복수 결제)

🔒 상태 변경 보안 규칙

  • 서버사이드 전용 유효성 검증
  • 클라이언트 파라미터 직접 수용 금지
  • 권한 없는 변경: HTTP 403
  • 유효하지 않은 전이: HTTP 422 + 에러코드
  • STL_PAID / INV_PAID: 변경 불가 처리
  • 상태 변경 API: PATCH /resource/{id}/status
  • PAY_PAID 중복 처리 방지 필수 (PG 웹훅)

🔔 알림톡 자동 발송 트리거

  • RES_RECEIVED → 예약 접수 안내
  • RES_CONFIRMED → 예약 확정 + 잔금 납부 안내
  • PAY_DEPOSIT_PAID → 예약금 입금 확인
  • PAY_BALANCE_REQUESTED → 잔금 납부 요청
  • PAY_PAID → 결제 완료
  • DEP_CONFIRMED → 출발 확정
  • 출발 D-3, D-1 → 리마인더 (스케줄)
  • RES_CANCELLED → 취소 확인
  • PAY_REFUNDED → 환불 완료
  • DEP_CANCELLED → 행사 취소 (전체 발송)
📋 v0.2 주요 검토 필요 항목
  • 예약만료(RES_EXPIRED) 기준 시간: 운영 환경에 맞게 설정값 정의 필요 (B2C: 30분, B2B 판매사: 별도 협의)
  • 잔금 미납 시 자동 취소 정책: PAY_BALANCE_REQUESTED 이후 미납 N일 → RES_CANCELLED 자동 전환 여부 및 기한 정의 필요
  • 알림톡 폴백 과금 주체: 알림톡 실패 → SMS 대체 발송 시 SMS 건당 과금 처리 및 정산 방식 확인 필요
  • TPL_MODIFY_REQUESTED 중 기존 템플릿 사용 가능 여부: 카카오 채널 정책 확인 후 AOS 동작 정의 필요
  • GOV_ACTIVE 예산 자동 소진 처리: 예산 한도 도달 시 할인코드 자동 비활성화 처리 로직 개발 명세 필요
  • PAY_REFUND_FAILED SLA: 환불 실패 후 수동 처리 기한(예: 영업일 3일) 정의 및 에스컬레이션 절차 수립 필요
  • EXPO_* 상태 생성 시점: SHARE_ACCEPTED 시 자동 EXPO_NOT_CONFIGURED 생성 로직 백엔드 명세 필요
✅ 상태값 추가 시 필수 체크리스트
  • 신규 상태의 이전/다음 전이 가능 상태를 명확히 정의했는가?
  • 신규 상태 진입 시 알림톡 발송이 필요한가? 승인된 TPL_APPROVED 템플릿이 존재하는가?
  • 신규 상태가 정산 금액 또는 정산 대상에 영향을 주는가?
  • DB 마이그레이션 스크립트 및 Enum/코드 테이블 업데이트가 준비되었는가?
  • 프론트엔드에서 해당 상태에 대응하는 배지/라벨 디자인이 정의되었는가?
  • 본 문서(AOS_상태값정의서)가 업데이트되고 팀 리뷰(기획+개발+QA)를 받았는가?
